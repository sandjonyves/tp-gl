Approche de Test Projet Propelize
Le choix dune ou plusieurs approches de test est crucial pour assurer une couverture efficace des
exigences tout en respectant les contraintes de temps, de ressources et de complexit du projet. Pour
le projet Propelize, plusieurs approches de test seront combines afin dassurer la fois la fiabilit
fonctionnelle, la scurit, et la qualit technique du code.
1. Black-box Testing (Test fonctionnel)
Pourquoi ?
Permet de vrifier que lAPI renvoie les bons rsultats en fonction des entres, sans se
soucier de la manire dont ces rsultats sont produits.
Exemples concrets :

2. White-box Testing (Test structurel)
Pourquoi ?
Utile pour tester la logique interne de certaines fonctions (par exemple : la validation
dentre, le traitement du token, les contrles de rle).
O ?
Fichiers middleware/, utils/, controllers/
3. Gray-box Testing
Pourquoi ?
Combine la vision externe et interne, pratique pour les membres de lquipe ayant accs au
code et aux spcifications.
Exemples :
Tester le bon fonctionnement dune route tout en analysant les conditions if, les
appels la base, etc.
4. Automated Testing (avec Jest & Playwright)
Pourquoi ?
Gain de temps
Excution continue via GitHub Actions
Documentation vivante des comportements attendus
Outils utiliss :
Jest + Supertest pour lAPI
Playwright pour le front-end (/login, /users, /vehicles)
5. Manual Testing (temporaire, complmentaire)
Pourquoi ?
Utile dans les premires phases de test ou pour explorer rapidement une nouvelle
fonctionnalit.
Outils :
Postman, Insomnia, navigateur
6. Regression Testing
Pourquoi ?
Sassurer quune nouvelle fonctionnalit (ex. ajout de la gestion des tokens) na pas
cass les anciennes (ex. GET /vehicles).
Comment ?
Rexcuter les anciens tests automatiquement aprs chaque ajout de code
7. Exploratory Testing (Hors porte principale)
Cette approche peut tre utilise ponctuellement, mais ne sera pas la base principale du
plan de test, car le projet suit un cahier des charges bien dfini avec des endpoints fixes.
Justification du choix des approches
Les approches ont t slectionnes pour assurer :
Une haute couverture fonctionnelle (Black-box)
Une matrise du comportement interne (White-box)
Une intgration dans un pipeline CI/CD (Automated testing)
